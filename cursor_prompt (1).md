# Промпт для Cursor: добавление панели управляющих воздействий оператора

## Контекст проекта

Я работаю над десктопным PyQt5-приложением **«Converter Simulator»** — обучающий симулятор для операторов сталеплавильного конвертера. Архитектура подробно описана в `ARCHITECTURE.md` (читай его перед началом работы — там расчётный движок, формулы, флаги этапов и связь с 3D).

Ключевое:
- Основная форма оператора — `OperForm.py`, класс `Ui_OperatorForm`, собирается **программно** (без `.ui`).
- Расчётный движок — 8 этапов, оркестратор `run_calculations()`, асинхронный запуск через `WorkerThread` → `GetScenarioExample()`.
- Все стили централизованы в `app_theme.py` (тёмная/светлая тема, тоggle без перезапуска).
- 3D-визуализация — `converter3d/widget.py`, единый API `update_state(dict)`.
- БД — MySQL (`regimdata`, сценарий хранит лимиты C/P/T).

## Цель доработки

В интерфейс оператора (закладка **«Схема»** в `OperForm` — это центральная панель формы, см. скриншот) добавить **панель ручного управления управляющими воздействиями**, чтобы оператор мог «крутить ручки» и менять параметры дутья **до запуска расчёта** (а в перспективе — пересчитывать сценарий с пользовательскими значениями).

### Минимальный набор управляющих воздействий (must-have)

| Параметр | Ед. изм. | Типовой диапазон | Где сейчас используется |
|----------|----------|------------------|--------------------------|
| **Объём дутья (расход O₂)** | м³ / м³·мин⁻¹ | 200–1200 м³/мин (для 100–350 т конвертера) | `blastCalcClicked()` → `TotalConsumptionOfBlastM3`, передаётся в 3D как `blastFlow` |
| **Время продувки** | мин | 12–22 мин | сейчас явно не задаётся — рассчитывается косвенно; нужно добавить как пользовательский ввод |
| **Интенсивность продувки** | м³/(т·мин) | 2.5–5.0 | производная от объёма дутья / массы металла / времени |
| **Положение фурмы над ванной** | м | 0.8–2.5 м | сейчас не моделируется — нужно добавить (влияет на дожигание CO и износ футеровки) |

### Желательно (nice-to-have, если найдёшь надёжные зависимости)

- **Расход извести / магнезиального флюса** (кг) — уже есть в `FluxeTable`, но «ручкой» удобнее.
- **Температура чугуна на заливке** (°C) — есть в БД, дублировать на панель крутилок.
- **Доля лома в металлошихте** (% от массы) — пересчитывает `m_лом / m_чугун`.

Реализуй их как новые приватные методы класса `Ui_OperatorForm`. Каждую формулу сопровождай docstring со ссылкой на источник из списка ниже.

## Формулы для новых управляющих воздействий

> Все формулы и коэффициенты — из открытых источников. Источники:
> - **[1]** Шаповалов А.Н. «Металлургия стали». Учебное пособие. — НФ НИТУ МИСиС, 2014.
> - **[2]** Шаповалов А.Н. «Технология и расчёт плавки стали в кислородных конвертерах». Методические указания к курсовой работе. — НФ НИТУ МИСиС, 2020.
> - **[3]** «Автоматизация конвертерного процесса производства стали». Учебное пособие КарГТУ, 2004.

### Ф-1. Удельная интенсивность продувки

```
i_уд = i / G_ст                       [м³ / (т · мин)]
```
- `i` — общий расход кислорода через фурму, м³/мин
- `G_ст` — масса выплавляемой стали (или металлошихты, для оценочного расчёта), т

**Диапазоны** (источник [1], [2]):
- Верхняя продувка: **2,5–7,0 м³/(т·мин)**, практический оптимум **3–5 м³/(т·мин)**.
- Донная: 3,0–7,0 м³/(т·мин).
- При `i_уд > 6` — режим переокисления, подсветка крутилки красным.
- При `i_уд < 2,5` или `> 5,5` — жёлтое предупреждение.

### Ф-2. Время продувки

```
τ_прод = V_дутья / i                  [мин]
```
- `V_дутья` — общая потребность в дутье на плавку, м³ (уже считается в `blastCalcClicked()` как `TotalConsumptionOfBlastM3`)
- `i` — расход кислорода, м³/мин

**Типовые значения** для конвертеров 100–350 т: **τ_прод ≤ 20–25 мин** (источник [3]).
Если пользователь задаёт `τ_прод` напрямую — пересчитывай `i = V_дутья / τ_прод` и обновляй соответствующее поле.

### Ф-3. Скорость истечения газа из сопла (адиабатическая)

```
ω_го = sqrt( 2 · K/(K-1) · R · T₂ · [ 1 − (P₁/P₂)^((K-1)/K) ] )      [м/с]
```
- `K` — показатель адиабаты, для O₂ принять **1,4** (двухатомный газ)
- `R = R₀ / M` — удельная газовая постоянная; `R₀ = 8,3143` Дж/(моль·К); `M(O₂) = 32·10⁻³` кг/моль → `R ≈ 259,8` Дж/(кг·К)
- `T₂` — температура газа в фурме, К (типично **293–323 К**, то есть 20–50 °C)
- `P₂` — давление перед соплом, Па (типично **1,0–1,4 МПа**)
- `P₁` — давление среды в полости конвертера, Па (~**1,2·10⁵ Па**)

**Типовая начальная скорость струи:** 400–500 м/с (сверхзвуковая, источник [1]).

### Ф-4. Импульс кислородной струи

```
i_г = ρ_г · V_г · ω_го / n            [Н]
```
- `ρ_г` — плотность газа при стандартных условиях; для O₂ принять **1,429 кг/м³**
- `V_г` — объёмный расход газа, м³/с (= `i` [м³/мин] / 60)
- `ω_го` — скорость на срезе сопла из Ф-3, м/с
- `n` — число сопел фурмы (типично 3–7; в учебном симуляторе — **5**)

### Ф-5. Модифицированный критерий Архимеда (для расчёта глубины внедрения)

```
Ar* = i_г / ( ρ_ж · g · h_c³ )         [безразмерный]
```
- `i_г` — импульс струи из Ф-4, Н
- `ρ_ж` — плотность жидкого металла, **7000 кг/м³**
- `g = 9,81 м/с²`
- `h_c` — высота сопла фурмы над уровнем спокойной ванны, м (это и есть наша «крутилка»)

### Ф-6. Глубина внедрения струи (лунки) в металл

Используй полуэмпирическую формулу через критерий Архимеда:

```
h_л / h_c = 0,78 · (Ar*)^0,35                # источник [1], аппроксимация регрессии
h_л = 0,78 · h_c · (Ar*)^0,35          [м]
```

Комментарий в коде: «Уравнение получено по экспериментальным данным водных моделей конвертера и обобщено в логарифмических координатах lg(h_л/h_c) = f(lg Ar*) — Шаповалов, МИСиС, 2014, гл. 2».

**Качественные закономерности** (для валидации):
- `h_л` растёт при уменьшении `h_c` (фурма ниже → глубже внедрение).
- `h_л` растёт при увеличении `V_г` (больше O₂ → глубже).
- `h_л` растёт при уменьшении диаметра сопла (выше скорость).
- При жёстком режиме (`h_c < 1 м`) — риск выбросов; кратковременно повышай `h_c` на 0,2–0,5 м для «сворачивания» шлака.

### Ф-7. Рекомендуемые диапазоны положения фурмы (зависят от массы плавки)

| Масса плавки, т | Начальное положение `h_c`, м | Рабочее (заглубл.), м |
|-----------------|-------------------------------|------------------------|
| 100–200 | 2,0–3,0 | 1,0–1,3 |
| 200–300 | 2,5–3,5 | 1,3–1,6 |
| 300–400 | 3,5–4,5 | 1,7–2,0 |

Источник [1]. Используй для функции `recommended_ranges()`:
- Жёлтая подсветка при выходе за «рабочее» окно для текущей массы шихты.
- Красная подсветка при `h_c < 0,8 м` или `h_c > 4,5 м` (физические ограничения).

### Ф-8. Корректировка теплового баланса при изменении положения фурмы

Положение фурмы влияет на **долю дожигания CO** в полости конвертера (которое в `HeatBalanceCalcClicked()` сейчас фиксировано как `Z ≈ 0,1`). Эмпирическая корреляция:

```
Z_CO = 0,10 + 0,08 · (h_c − h_c_оптим) / h_c_оптим
Z_CO = clip(Z_CO, 0,05, 0,30)
```
- `h_c_оптим` — середина рабочего диапазона для текущей массы (Ф-7)
- При высокой фурме `Z_CO` растёт (CO успевает дожигаться над ванной) — это даёт **дополнительное тепло** через `Q_CO = 101 · g_CO · Z_CO · 1000` (формула уже есть в `HeatBalanceCalcClicked()`, см. ARCHITECTURE.md → раздел 6).

Комментарий: модель упрощённая, основана на качественной закономерности из [1] (выше фурма → больше площадь контакта вторичной реакционной зоны с газом).

### Ф-9. Корректировка износа футеровки (расширение `deoxCalc()`)

Текущая формула износа в `deoxCalc()`:
```
Износ = 4,112·10⁻⁶ · T · (lim_MgO · MgO_шлака)
```
Добавь поправочный коэффициент за положение фурмы и интенсивность продувки (жёсткий режим увеличивает износ):

```
k_лк = 1 + 0,15 · max(0, (i_уд − 4,5) / 4,5)         # перегруз по интенсивности
k_фурма = 1 + 0,20 · max(0, (1,3 − h_c) / 1,3)        # фурма слишком низко
Износ_скорр = Износ · k_лк · k_фурма
```

Коэффициенты 0,15 и 0,20 — оценочные, основаны на качественных данных [1] (раздел про стойкость футеровки). Прокомментируй это в коде как «учебная модель, требует калибровки по реальным данным цеха».

### Ф-10. Связь с расчётом дутья (модификация `blastCalcClicked()`)

Если оператор задал **объём дутья** `V_user` через крутилку — использовать его как `TotalConsumptionOfBlastM3`, иначе оставлять текущий расчёт.

Если оператор задал **интенсивность** `i_user` — пересчитать время:
```
τ_прод = TotalConsumptionOfBlastM3 / i_user
```

Если оператор задал **время** `τ_user` — пересчитать интенсивность:
```
i = TotalConsumptionOfBlastM3 / τ_user
i_уд = i / G_металлошихты
```

Два из трёх параметров (`V`, `i`, `τ`) — независимые; третий вычисляется. UI должен показывать «вычисленный» параметр серым/курсивом.

### Ф-11. Передача в 3D-сцену

Расширь словарь `update_state(dict)`:
```python
{
    'blastFlow': i,                # м³/мин — для анимации интенсивности дутья
    'lanceHeight': h_c,            # м — для анимации положения фурмы
    'penetrationDepth': h_л,       # м — для визуализации лунки в ванне
    'reactionZoneActive': True,    # подсветка реакционной зоны
}
```

В `converter3d/converter.html` (Three.js) добавь:
- Меш фурмы, позиция Y которого = `bath_top_y + lanceHeight * scale`.
- Полупрозрачный конус «лунки» от носика фурмы вглубь ванны глубиной `penetrationDepth * scale`.
- Glow-эффект (PointLight оранжево-белый) в центре лунки при `reactionZoneActive = True`.

## Технические требования

### 1. Расположение и компоновка

- Добавить новую группу `QGroupBox("Управляющие воздействия")` в **центральную колонку** формы оператора, **над** блоком «Последовательность расчётов» (или в отдельной вкладке «Ручное управление» рядом со «Схема / Окисление / Шлак / …» — выбери лучший вариант по UX и обоснуй в комментарии).
- Каждая «крутилка» = композитный виджет:
  - `QLabel` (название параметра + ед. изм.)
  - `QDial` **или** `QSlider` (горизонтальный) — выбери исходя из эргономики, у `QDial` более «приборный» вид
  - `QDoubleSpinBox` для точного ввода + синхронизация с диалом через `valueChanged`
  - Маленький `QLabel` справа с текущим значением и единицами
- Минимум 4 крутилки в одну строку через `QGridLayout` (2×2) — на большом экране, и адаптивно сворачивать в 1×4 на узком (использовать `QSplitter` или `QScrollArea`).

### 2. Логика и связь с расчётом

- Создать отдельный модуль `control_inputs.py` с классом `ControlInputsPanel(QWidget)`:
  - Сигнал `controls_changed = pyqtSignal(dict)` — испускается при изменении любой крутилки (с дебаунсом 200 мс через `QTimer.singleShot`).
  - Метод `get_values() -> dict` — возвращает текущие значения всех параметров.
  - Метод `set_values(dict)` — программно установить значения (для загрузки из сценария).
  - Метод `reset_to_defaults()` — кнопка «Сброс».
- В `OperForm.py`:
  - Создать экземпляр `ControlInputsPanel`, подключить к `controls_changed` слот `_on_controls_changed(values)`.
  - В этом слоте: сохранять значения в атрибуты класса (`self.userBlastVolume`, `self.userBlowTime`, `self.userLanceHeight`, …).
  - **Модифицировать `blastCalcClicked()`**: если пользователь задал свои значения — использовать их вместо рассчитанных; рассчитанные значения показывать как «рекомендуемые» (placeholder/serый текст в `QDoubleSpinBox`).
  - В `MaterialBalanceCalcClicked()` и `HeatBalanceCalcClicked()` — учесть пользовательский объём дутья при пересчёте газового и теплового баланса.
- Добавить кнопку **«Применить и пересчитать»** — запускает `run_calculations()` с пользовательскими параметрами (через `WorkerThread`, как сейчас).

### 3. Визуальная обратная связь

- При изменении объёма дутья → немедленно обновить 3D-сцену: `self.converter3d.update_state({'blastFlow': new_value})`.
- При изменении положения фурмы → передать в 3D как новый ключ `'lanceHeight'` (потребует расширения `converter3d/widget.py` и Three.js-сцены — добавь, опираясь на текущий контракт `update_state(dict)`).
- Подсветка крутилки **жёлтым** (через QSS), если значение выходит за рекомендуемый диапазон (рекомендуемый диапазон рассчитывается из массы шихты и марки стали — добавь утилиту `recommended_ranges(scenario_data) -> dict`).
- Подсветка **красным**, если значение нарушает физические ограничения (например, интенсивность > 6 м³/(т·мин) — режим переокисления).

### 4. Стилизация

- Использовать **только токены и палитру из `app_theme.py`** — не вводить хардкод цветов. Если нужны новые токены (например, `control_dial_track`, `control_dial_handle`) — добавить их в `app_theme.py` для обеих тем (dark/light).
- Подключиться к сигналу `ThemeManager.theme_changed` чтобы переключение темы работало без перезапуска.
- Шрифты — те же, что в остальной форме (через `app_theme`).

### 5. Сохранение и загрузка пресетов

- Добавить две маленькие кнопки в углу панели: **«Сохранить пресет»** и **«Загрузить пресет»**.
- Пресет = JSON-файл со значениями всех крутилок + ID сценария + timestamp.
- Папка по умолчанию: `presets/` в корне проекта (создавать при первом сохранении).
- Это пригодится для воспроизводимости обучения операторов.

### 6. Протокол обучения

- Каждое изменение параметра логировать в `Protokol` (глобальная строка протокола, см. `ARCHITECTURE.md` → «Конфигурация»):
  - `[HH:MM:SS] Оператор изменил «Объём дутья» с 850 → 920 м³/мин`.
- Это важно для post-mortem анализа действий обучаемого.

## Что НЕ ломать

- Существующие 8 этапов расчёта и порядок их вызова в `run_calculations()`.
- Guard-логику кнопок этапов (нельзя перейти к следующему без завершения предыдущего).
- LED-индикаторы и таймер `_refresh_stage_leds()`.
- Кнопку «ЗАПУСТИТЬ ВСЕ ЭТАПЫ» — она должна продолжить работать со значениями из крутилок, если те заданы; иначе — с расчётными.
- Сценарии из БД (`regimdata.scenario`) — пользовательские значения **не должны** записываться обратно в БД.
